# Problem Statement

[← Home](Home.md) | [Next: AI Usage Survey](ai-usage-patterns.md)

Software engineers are adopting AI tools quickly, but most usage remains informal, local, and difficult to control at scale.

The central issue is not whether AI can generate code. It can.

The harder problem is whether AI-assisted work can remain coherent across:

- repeated changes
- larger enhancements
- existing codebases
- team practices
- architecture decisions
- documentation and validation

---

## The Practical Problem

Developers commonly use AI to:

- fix bugs
- make small changes
- add features
- refactor code
- generate documentation
- start new applications

These tasks work well when they are small, local, and clearly scoped.

They become harder when they require coordination across multiple files, layers, decisions, or team conventions.

---

## What Goes Wrong

Across current AI workflows, several problems repeat:

- context is reconstructed every time
- architecture is inferred rather than explicit
- decisions are hidden inside prompts or generated code
- related files may be missed
- validation is mostly reactive
- documentation becomes disconnected from implementation
- team practices are not consistently applied

---

## Why This Matters

As AI use moves from isolated assistance to real software delivery, the concern shifts from speed to reliability.

The question becomes:

```text
How do we preserve engineering discipline while using AI to accelerate work?
```

---

## Scope of This Repository

This repository explores that problem space.

It begins with a survey of how engineers currently use AI tools, then identifies recurring gaps, and only later introduces one possible structured approach.
